What means ...

Deploy: "to install, test and implement an application or computer system."
Continuous Integration: "to integrate your work constantly with your team members."
Continuous Delivery: "provide your work constantly for review by the team or client."
Continuous Deployment: "deploy your work constantly into client's server."
=> Continuous Deployment becomes a reflex after having been a discipline."

Here's how to start

  • Coding standards (lint)
  • Everything in code (comments)
  • Development environment should be like production environment
  • Automated tests

Why version control is important ?

  • Colaboration (and avoiding shouting through the office that you are curently working on file xyz...)
  • Storing versions
  • Restoring previous versions
  • Enable offline work (git only)

Local and remote repositories

  • A local repository resides on your local computer
  • A remote repository is located on a remote server. Used by teams to share & exchange data: they are the common base. Default name is origin.

2 ways to get a local repository : git init or git clone

Basic commands

  • git pull origin master
  • git branch new_branch + git checkout new_branch = git checkout -b new_branch
  • git add file_name
  • git commit -m"feat: add file_name"
  • git push origin master
  • git status
  • git log

Commit best practices

    git commit -m "doc: add usage in README.md"
  • Commit related changes (git add -p helps a lot)
    • 2 differents bug fixes => 2 different commits
    • small commits => easier for team members to understand
  • Commit often
    • helps keeping small commits
    • helps integrating changes regularly
    • avoid merge conflicts

Commit best practices

    git commit -m "doc: add usage in README.md"
  • Don't commit half done work
  • Test before commit
  • Write good commit messages
    • use the imperative, present tense ("change" not "changed" of "changes")
    • use a commit comment prefix in ("feat:", "fix:", "ref:", "doc:")

Branching best practices

  • master branch, as the production branch is the mainline
    • should be protected from manual push and merge
  • Know the current branch git branch -a
  • Tag releases on master branch.

Branching best practices

  • Only merge with neighbors (featureX -> dev -> master)
  • Create branches from master/develop at first. The further from master, the crazier

Git workflow

  1. Pull to update your local master
  2. Check out a feature branch
  3. Do work in your feature branch, committing early and often
  4. Rebase frequently to incorporate upstream changes
  5. Interactive rebase (squash) your commits
  6. Merge your changes with master/develop
  7. Push your changes to the upstream

Merge requests on gitlab ...

  • ... or pull requests on github
  • Current benefits :
    • code review (comment for anyone, ask for change, education, guidelines)
    • reduces conflicts
    • make it visual
    • continuous delivery
  • Long term benefits :
    • automated tests once the continuous integration is set up => better stability
    • link with jira tickets

Dont's

  • Rush & use reset without knowing what you do
  • Use git merge : use merge requests instead (even for hotfixes)
  • Commit configuration files
    • might change from env to env & contain passwords and creds
  • Commit large binary files
  • Commit anything that can be generated from other things that were commited
    • use .gitignore a lot to avoid that
    • except for package-lock.json ! details here

Do's, Tips & tricks

  • Never ever panic !
    • once commited the work can't be lost for at least 2 weeks
    • before, attempting to find lost commits, be sure you will not lose any current work (git commit or git stash, but name your stashes !)
    • git reflog is (sometimes) your friend
  • Keep a clean history : git rebase -i HEAD~12 helps a lot

Tips & tricks

  • Understand, what is HEAD, to understand error messages !
    • reference to the current commit in the current branch, only 1 HEAD at a given time
    • if you're not on the latest commit, HEAD is a point to a prior commit in history (git checkout commit_id), its called detached HEAD.
  • Pull with git pull --rebase to keep a linear history
  • Validate your repo is sane git fsck ?
